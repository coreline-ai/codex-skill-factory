## 문서 상세 분석 요약

- 목표는 Codex IDE/CLI 자체를 바꾸지 않고 hook과 별도 CLI만 추가하는 경량 자동화다.
- 데이터 흐름은 `UserPromptSubmit` → `prompts.jsonl`, `Stop` → `turns.jsonl`, CLI 분석 → `candidates.json`/`report.md`, 사용자 승인 → `.codex/skills/<name>/SKILL.md` 순서다.
- MVP 핵심은 로깅 안정성, 민감정보 마스킹, 규칙 기반 후보 생성, 보고서/미리보기/생성 명령이다.
- 문서의 고도화 Phase 3~5는 향후 범위이며 이번 구현에서는 구조 확장 가능성만 유지한다.
- Skill 후보 판단 기준상 반복 빈도와 절차/검증 가능성이 중요하므로 `scan`은 최소 빈도 기준을 적용해야 한다.
- AGENTS.md와 Skill의 역할 구분이 명시되어 있어, 이번 구현은 항상 적용되는 저장소 규칙을 만들지 않고 특정 반복 요청용 Skill 생성에 집중한다.

## 잔여 리스크 / 후속 과제

- 실제 Codex hook payload 스키마가 환경별로 다를 수 있어 운영 중 발견되는 payload key는 hook 스크립트에서 보강해야 한다.
- 현재 후보 생성은 키워드/빈도 기반이므로 Phase 3 고도화 항목인 임베딩 기반 클러스터링은 후속 개발로 남긴다.
- 생성된 Skill 승인/무시 워크플로우는 CLI 수동 실행 기반이며, 대시보드/UI는 후속 범위다.
