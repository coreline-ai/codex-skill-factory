## 개발 목적

기존 MVP 구현 위에 `docs/codex_prompt_skill_factory_dev_doc.md`의 고도화 범위인 임베딩 기반 유사도 분석, 성공률 분석 hook, VS Code 대시보드, 승인/무시 UI를 실제 사용 가능한 로컬-first 기능으로 확장한다.

## 개발 범위

- 프롬프트 텍스트를 벡터화해 유사 요청 클러스터 후보를 생성하는 분석 모듈 추가
- `scan` CLI에서 규칙 기반 후보와 유사도 기반 후보를 함께 생성하도록 확장
- `PostToolUse` hook 및 `Stop` hook 보강으로 명령 성공/실패, 테스트/lint 신호, 변경 파일 수, 반복 수정 신호를 저장
- 로그 기반 성공률/운영 지표를 계산하는 analytics 모듈 및 CLI 추가
- 후보 승인/무시/상태 관리 CLI와 인터랙티브 review UI 추가
- VS Code에서 열어볼 수 있는 정적 HTML 대시보드 및 JSON 데이터 생성
- `.vscode/tasks.json`으로 대시보드 재생성/스캔 작업 제공
- 관련 단위 테스트와 스모크 검증 추가

## 고도화 전문가 리뷰

- 임베딩 분석은 MVP의 키워드 규칙을 대체하지 말고 보완해야 한다. 초기 버전에서는 무거운 모델 대신 표준 라이브러리 기반 로컬 벡터화로 결정론적 테스트 가능성을 확보하고, 이후 `sentence-transformers`를 optional backend로 붙일 수 있는 구조가 적합하다.
- 성공률 분석은 hook payload가 환경마다 다를 수 있으므로 payload 원문 의존을 최소화하고 `exit_code`, `status`, `command`, `tool_name`, `transcript` 등 복수 키를 관대하게 해석해야 한다.
- 승인/무시는 생성 행위와 분리해야 한다. `pending_review`, `approved`, `ignored`, `created` 상태를 후보 JSON에 유지하고, CLI가 상태 전이를 담당해야 한다.
- VS Code 대시보드는 처음부터 extension으로 만들기보다 정적 HTML + `.vscode/tasks.json` 조합이 비용 대비 효과가 좋다. 대시보드에는 직접 쓰기 대신 복사 가능한 CLI 명령을 제공한다.
- 후보 판단 기준은 빈도만이 아니라 최근성, 성공률, 검증 신호, 반복 수정 신호를 함께 보여줘야 사용자가 승인 여부를 판단할 수 있다.

## 잔여 리스크 / 후속 과제

- 현재 임베딩은 lightweight 로컬 벡터화다. 더 높은 의미 유사도 품질이 필요하면 optional backend로 `sentence-transformers`를 추가한다.
- 정적 HTML 대시보드는 보안상 파일을 직접 수정하지 않고 CLI 명령을 제공한다. 완전한 버튼 기반 승인을 원하면 VS Code Extension 또는 로컬 서버가 필요하다.
- 실제 Codex hook payload는 버전별 차이가 있을 수 있으므로 운영 로그를 보며 payload key 추출 규칙을 보강해야 한다.
